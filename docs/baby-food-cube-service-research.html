이유식 큐브 재고관리 서비스 리서치

결론: 추천 기본안은 GitHub Pages를 데이터베이스처럼 쓰는 구조가 아니라, 정적 PWA에 IndexedDB를 붙인 local-first 앱입니다. GitHub Pages나 Cloudflare Pages에는 UI만 올리고, 아이 식단·재고·반응 데이터는 기본적으로 로컬에 둡니다.

추천 아키텍처

1단계 MVP

  • 정적 PWA: GitHub Pages 또는 Cloudflare Pages 배포.
  • Primary data: IndexedDB.
  • Backup: JSON/CSV export, optional encrypted backup.
  • UI tabs: 오늘, 식단표, 큐브, 재료, 조합, 기록.
  • AI: 직접 쓰기 대신 구조화 변경안 preview/import.

공유가 필요해질 때

  • 가장 단순: Firebase Firestore.
  • 개발자 제어권 중시: Cloudflare Pages + Workers + D1.
  • 브라우저 local-first sync 중시: Dexie Cloud.

요구 기능 대응

요구설계
재고관리CubeLot에 제조일, 소비기한, 남은 큐브 수, g/ml per cube 저장.
수량 미달 확인StockRule로 최소 큐브 수와 주간 식단 예약분 반영.
아직 안먹어본 식재료 등록Ingredient.status를 not_tried/planned/testing/tolerated로 관리.
큐브 조합 관리Combination에 필요한 큐브, 단계, 질감, 알레르기 태그 저장.
1주일 식단 관리MealPlanSlot으로 날짜, 식사 종류, 조합, 도입 식재료 관리.
UI + AI 업데이트모바일 PWA + AI 변경안 preview/import.
GitHub Pages 확인앱 셸 배포에는 적합, 원본 개인정보 데이터 배포에는 부적합.

AI 업데이트 파이프라인

  1. 사용자가 “소고기 큐브 6개 만들었어”라고 요청.
  2. AI가 구조화 변경안을 생성.
  3. validator가 SKU, 단위, 수량, 권한, 음수 재고, 중복 요청을 검사.
  4. 애매하거나 큰 변경은 확인 요청.
  5. append-only event로 저장.
  6. 현재 재고와 식단을 재계산.
  7. 감사 로그와 rollback handle을 남김.

GitHub 기반을 꼭 쓰는 경우

GitHub은 데이터베이스가 아니라 감사 가능한 변경 로그와 PR 시스템으로 사용합니다. AI는 branch와 PR만 만들고, 검증 workflow가 통과해야 merge합니다. raw baby/family data는 public/에 배포하지 않습니다.

최종 추천

처음 만들 때는 local-first PWA로 시작하는 것이 가장 안전하고 단순합니다. 가족 간 실시간 공유가 실제 요구가 되면 Firestore 또는 Dexie Cloud로 sync를 붙이고, 운영/비용 통제를 더 원하면 Cloudflare Workers + D1로 갑니다.

GitHub Pages + AI commit 방식은 앱 배포와 변경 감사에는 좋지만, 아이 식단 데이터 원본 저장소로는 기본값이 아닙니다.

한계와 확인 필요 사항

  • 이 보고서는 desk research 기반입니다. 실제 부모 사용성은 빠른 기록과 AI 변경 확인 사이의 마찰을 사용자 인터뷰나 프로토타입 테스트로 확인해야 합니다.
  • 앱은 식재료 도입과 반응을 기록할 수는 있지만 알레르기나 영양 상태를 진단하면 안 됩니다. AI 영양 분석이나 의료적 조언을 넣는다면 별도 법률·의료 검토가 필요합니다.
  • 암호화 백업은 추천 방향이지만, 실제 비밀번호 복구 정책과 휴대폰 성능에서의 KDF 비용은 구현 전에 측정해야 합니다.
  • GitHub 기반 업데이트는 감사/배포 워크플로로는 유용하지만 실시간 동기화 DB가 아닙니다. 가족 다중기기 동기화가 핵심 요구가 되면 Firestore, Dexie Cloud, Cloudflare D1 같은 동기화 계층을 별도로 선택해야 합니다.
